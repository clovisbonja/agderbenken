Introduksjon

Prosjektoversikt

Sørblikket er et digitalt journalistikkverktøy som gjør det enkelt å følge med på hva Agder-regionens stortingsrepresentanter faktisk gjør — ikke bare hva de lover.

Formål

Stortingets åpne data er tilgjengelige på data.stortinget.no, men kun som rå XML-filer uten søkefunksjon, filtrering eller visuell presentasjon. Sørblikket bygger bro mellom disse rådata og en forståelig fremstilling for journalister og innbyggere i Agder. Appen henter, tolker og presenterer data automatisk — slik at brukeren kan fokusere på det politiske innholdet.

Prosjektet er primært beregnet på bruk av lokalaviser og nyhetsredaksjoner i Agder-regionen som vil ha effektiv tilgang til faktainformasjon om representantenes virksomhet på Stortinget.

Målgruppe

Journalister
Regionalavisreportere som trenger rask tilgang til faktabasert informasjon om politisk aktivitet, voteringer og valgløfter.
Innbyggere i Agder
Folk som ønsker å forstå hva «sine» representanter gjør på Stortinget og om de stemmer i tråd med det de lovet i valgkampen.
Studenter
Statsvitenskap- og informasjonssystemstudenter som studerer politisk representasjon eller webapplikasjonsarkitektur.
Regionale organisasjoner
Næringsforeninger, fagforeninger og interessegrupper med behov for løpende innsikt i saker av særlig betydning for Agder.

Sentralt arkitekturprinsipp: ingen server-proxy

Alle API-kall går direkte fra brukerens nettleser. Det er ingen mellomserver som Sørblikket drifter. Appen er ren statisk frontend og kan hostes gratis på Netlify, Vercel eller GitHub Pages. Stortingets API er åpent for alle og støtter CORS, noe som gjør direkteoppkobling mulig og forsvarlig. Appen fungerer så lenge Stortingets API er oppe — uavhengig av driftkostnader for egne servere.

Dataflyten i fire steg

1
Stortinget API
Åpent XML-API på data.stortinget.no. Ingen API-nøkkel. Inneholder saker, voteringer, representanter, biografier og komiteer.
→
2
fetch() i nettleseren
Direkte HTTP GET fra React-appen. AbortController for kansellering. setInterval trigger re-fetch hvert 5. minutt.
→
3
XML-parsing
DOMParser konverterer XML-streng til Document. Namespace-bevisste hjelpefunksjoner i xml.ts ekstraherer data til TypeScript-objekter.
→
4
React UI
Kategoriserte TypeScript-objekter lagres i useState. React re-rendrer kun endrede deler. Brukeren ser oppdatert innhold uten sideinnlasting.

Teknologi

Teknisk stack

Sørblikket er bygget på et moderne React-fundament med TypeScript gjennomgående. Teknologivalgene er tatt for å holde appens kompleksitet lav og driftsbehovet minimalt.

Teknologi Versjon Rolle og begrunnelse
React 19 UI-rammeverk. Komponentbasert arkitektur gjør det enkelt å isolere ansvar per side. React 19 gir bedre ytelse og forbedret concurrent rendering.
TypeScript 5.x Typesikkerhet gjennom hele kodebasen. Spesielt viktig ved XML-parsing der datastrukturer lett kan være uforutsigbare. Typer er definert i src/types/sak.ts.
Vite 6.x Byggverktøy og utviklingsserver. Raskere enn Create React App og Webpack. ESM-basert hot module replacement gir rask utviklingssløyfe.
React Router v7 Klient-side routing. Alle syv sider navigeres uten fullsideinnlasting. Konfigurert med createBrowserRouter og RouterProvider.
Supabase 2.x JS SDK PostgreSQL-as-a-service. Brukes kun på Partiprogrammer-siden for valgløfte-tabellen. Resten av appen er fullstendig serverløs.
DOMParser Web API (innebygd) XML-parsing direkte i nettleseren. Ingen ekstra bibliotek. Bruker getElementsByTagNameNS("*", tagname) for namespace-uavhengig parsing.
CSS (egendefinert) — En CSS-fil per side under src/styles/. Ingen Tailwind eller CSS-in-JS. CSS-variabler for temabytte. Gir full kontroll uten abstraksjonslag.

Bevisste valg om hva som ikke brukes

Ingen server-proxy
Alle API-kall direkte fra nettleseren. Ingen Node/Express/Deno backend som Sørblikket drifter. Reduserer driftskompleksitet og kostnad til null for API-laget.
Ingen Redux / Zustand
State håndteres med innebygd useState og useEffect per side. Appen er ikke stor nok til å rettferdiggjøre et globalt state-bibliotek.
Ingen React Query / SWR
Datafetching og caching er implementert manuelt i useStortingetData.ts. Holder avhengighetslisten kort og gir eksplisitt kontroll over oppdateringslogikk.
Ingen Tailwind CSS
Egendefinerte CSS-filer per side gir bedre lesbarhet og fleksibilitet for et prosjekt med tydelig visuell identitet. CSS-variabler håndterer temabytte.

Sesjon-ID-beregning

Stortinget organiserer arbeidet sitt i sesjoner som løper fra oktober til september. En sesjon navngis etter det året den starter, altså høsten. Sesjon 2025 starter i oktober 2025 og avsluttes i september 2026. Sørblikket beregner gjeldende sesjon-ID fra systemklokkens dato:

Oktober–desember: Ny sesjon starter — sesjon-ID = inneværende år (f.eks. 2025) Januar–september: Inneværende sesjon startet forrige høst — sesjon-ID = forrige år (f.eks. 2025 for januar–september 2026)

Logikken er implementert i src/utils/dato.ts og eksportert som en konstant som brukes i alle hook-kall mot Stortinget API.

Valgløfte-chatbot

Chatbot-arkitektur

Partiprogrammer-siden (/parti) inneholder en valgløfte-chatbot. Det er viktig å forstå nøyaktig hva denne er — og hva den ikke er.

Chatboten er ikke generativ KI. Den kaller ikke Claude API, OpenAI eller noen annen stor språkmodell i sanntid. Den er et nøkkelordssøk mot en Supabase-database som inneholder forhåndsuttrukne valgløfter. Brukeren skriver et spørsmål; appen søker etter relevante løfter ved hjelp av ilike-filtrering og lokal relevansskåring.

Slik fungerer chatboten — steg for steg

1
Brukerinndata
Brukeren skriver en fritekstspørring, f.eks. «hva lover Høyre om barnehage?»
→
2
Tokenisering
Inndataen deles opp i enkeltord (tokens). Stoppord som «hva», «om» og «er» filtreres bort.
→
3
Supabase-søk
For hvert meningsbærende token sendes en ilike-forespørsel mot valglofte-tabellen i Supabase. Filtrerer på parti og/eller kategori hvis oppgitt.
→
4
Relevansskåring
Resultatene skåres lokalt i nettleseren basert på antall treff og nærhet til søkeordet. De mest relevante løftene vises øverst.

Supabase-tabellstruktur

Tabell Kolonner Innhold
valglofte id, parti, tekst, kategori Uttrukne valgløfter fra partiprogrammer. Populert offline av extract-promises.mjs.
agder_lofte id, representant, tekst, kilde Skriftlige spørsmål fra Agder-representanter. Populert offline av extract-agder-promises.mjs.

Rolleavgrensning: Claude AI brukes kun offline

Claude AI (Anthropic) brukes utelukkende i det manuelle offline-skriptet extract-promises.mjs for å tolke partiprogrammer og identifisere valgløfter. Denne prosessen kjøres én gang av utvikleren og lagrer resultater i Supabase. Deretter er Claude AI ikke involvert i appens kjøretid — verken i chatbot-svaret eller noen annen del av appen som brukeren ser.

Oppsummering: Claude AI (offline) → leser PDF/HTML-partiprogrammer → identifiserer løfter → lagrer i Supabase. Brukeren spør chatboten → nettleseren søker i Supabase med ilike → lokalt rangerer resultater → viser relevante løfter. Ingen generativ KI i sanntid.

Tilgang og sikkerhet

Supabase-klienten i src/lib/supabase.ts er konfigurert med anon-nøkkelen (offentlig lesenøkkel). Denne nøkkelen er ikke hemmelig — den tillater kun lesing av tabeller der Row Level Security er konfigurert til å tillate det. Nøkkelen er lagret i miljøvariabelen VITE_SUPABASE_ANON_KEY.

Offline-verktøy

Scripts og datainnsamling

Datautvinningsskriptene er ikke en del av React-appen og kjøres aldri automatisk. De kjøres manuelt av utvikleren, typisk én gang per valgperiode eller ved behov for oppdatering.

Disse skriptene er ikke del av appen brukerne ser. De er Node.js-skript (.mjs) som kjøres fra terminalen. De kaller Claude API og Stortingets API, og lagrer resultater i Supabase-databasen. Etter at de er kjørt, er databasen oppdatert og appen leser fra den som vanlig.

Aktive skript

Skript Funksjon Målkilde
extract-promises.mjs Laster ned partiprogrammer (PDF og HTML), sender dem til Claude AI for identifisering av valgløfter, og lagrer strukturerte løfter i Supabase-tabellen valglofte. Supabase: valglofte
extract-agder-promises.mjs Henter skriftlige spørsmål fra Agder-representanter via Stortingets API, sender til Claude AI for oppsummering, og lagrer i agder_lofte-tabellen. Supabase: agder_lofte
retry-failed.mjs Gjentar ekstraksjon for partier der forrige kjøring feilet. Leser feillogg fra forrige kjøring. Supabase: valglofte
retry-missing.mjs Finner partier som mangler løfter i databasen og prøver ekstraksjon på nytt. Supabase: valglofte
retry-sp-r.mjs Spesialisert retry for Senterpartiet og Rødt, som kan ha ikke-standard programformater. Supabase: valglofte

SQL-skjema

Databaseskjemaet for Supabase er dokumentert i agder-lofte-tabell.sql. Dette SQL-skriptet oppretter agder_lofte-tabellen og tilhørende Row Level Security-regler. Kjøres direkte i Supabase SQL Editor ved oppsett.

Skript som er fjernet

Følgende skript finnes ikke lenger og skal ikke omtales som aktive funksjoner:
  • match-promises.mjs — var koblingen mellom valgløfter og konkrete stortingssaker. Slettet.
  • update-status.mjs — sporet om et valgløfte var innfridd, brutt eller under behandling. Slettet.
Som konsekvens: det finnes ingen kobling mellom valgløfter og stortingssaker i appen, og det finnes ingen innfridd/brutt-statussporing. Valgløfte-chatboten viser kun rå løftetekst fra partiprogrammene.

Prototype-side som er fjernet

Prototyper.tsx var en eksperimentell side som ble slettet under utviklingen. Den er ikke lenger registrert som en rute og skal ikke nevnes i dokumentasjon rettet mot brukere.

Fuzzy matching

Kategoriseringsmotor

Stortingssaker har titler, men ingen forhåndsdefinerte temaetiketter. Sørblikket kategoriserer saker automatisk ved hjelp av fuzzy matching basert på Levenshtein-distanse i src/lib/categorizationEngine.ts.

Hva er Levenshtein-distanse?

Levenshtein-distanse er et mål på hvor mange enkeltbokstav-operasjoner (innsetting, sletting eller erstatning) som trengs for å gjøre én streng til en annen. Distansen mellom «klima» og «klime» er 1 (én erstatning). Distansen mellom «helse» og «velferd» er større og indikerer at ordene er ulike konsepter. Fuzzy matching bruker dette til å finne saksord som er nær et nøkkelord, selv om de ikke er identiske — f.eks. på grunn av bøying eller skrivefeil.

Scoringsalgoritme

1
Tokeniser sakstittel
Del tittelen opp i enkeltord. Konverter til lowercase. Fjern tegnsetting.
→
2
For hvert tema
Iterer over alle åtte temaer og deres nøkkelordslister fra temaer.ts.
→
3
Beregn distanse
For hvert saksord og hvert nøkkelord: beregn Levenshtein-distanse. Eksakt treff gir høyest skår.
→
4
Velg beste tema
Temaet med høyest samlet skår vinner. Hvis ingen skår overstiger terskelverdien: tema = «Annet».

Temaer og nøkkelord (utvalg)

Tema Eksempel-nøkkelord

Klima og miljø klima, miljø, utslipp, fornybar, CO2, energi, natur, biologisk mangfold

Begrensninger

Fuzzy matching er ikke perfekt. Saker med generelle titler som «Representantforslag fra stortingsrepresentantene om ...» kan være vanskelige å kategorisere uten at selve lovteksten analyseres. I slike tilfeller faller saken i kategorien «Annet». Kategoriseringen er et hjelpemiddel for navigasjon og filtrering — den erstatter ikke redaksjonell vurdering.

Dokumentsammendrag

Dette dokumentet har gjennomgått Sørblikkets arkitektur i sin helhet. Nedenfor er de viktigste punktene oppsummert.

Kjernearkitektur

  • Ren klientside-applikasjon — ingen server-proxy, ingen backend driftet av Sørblikket
  • Alle kall til Stortingets API (data.stortinget.no) skjer direkte fra nettleseren via fetch()
  • All API-respons fra Stortinget er XML — ingen JSON. Parsing via DOMParser med namespace-bevisste hjelpefunksjoner
  • Auto-refresh hvert 5. minutt med setInterval. Cleanup via clearInterval og AbortController
  • Sesjon-ID beregnes fra systemklokkens dato: oktober–desember gir ny sesjon, januar–september gir forrige års sesjon

Supabase brukes kun til én ting

  • Supabase er i bruk kun på /parti-siden for valgløfte-chatboten
  • Tabellene valglofte og agder_lofte populeres offline av manuelt kjørte skript
  • Chatboten er nøkkelordssøk med ilike-filtrering — ikke generativ KI i sanntid
  • Claude AI brukes kun i det offline skriptet extract-promises.mjs, aldri i appen selv

Hva som er fjernet

  • Prototyper.tsx er slettet og ikke lenger en aktiv rute
  • match-promises.mjs er slettet — det finnes ingen kobling mellom valgløfter og stortingssaker
  • update-status.mjs er slettet — det finnes ingen innfridd/brutt-statussporing

Nøkkelfiler å kjenne til

src/lib/categorizationEngine.ts Fuzzy matching med Levenshtein-distanse for kategorisering av saker src/hooks/useStortingetData.ts Sentral datafetching-hook med auto-refresh og XML-parsing src/utils/xml.ts DOMParser-hjelpefunksjoner for namespace-uavhengig XML-parsing src/utils/dato.ts Sesjon-ID-beregning basert på gjeldende dato src/lib/supabase.ts Supabase-klient med anon-nøkkel fra miljøvariabel src/config/temaer.ts Åtte temaer med nøkkelordslister — brukt av kategoriseringsmotoren extract-promises.mjs Offline-skript: laster ned partiprogrammer, kaller Claude AI, lagrer i Supabase
